Day 1

최소 하네스: Day 1에 만들 것

Claude Code도 여기서 시작했다. 당신의 시스템도 여기서 출발해야 한다.

while (true) { // 1. 시스템 프롬프트 + 사용자 메시지 → API 호출 const response = await callLLM(messages); // 2. 응답에서 tool_use 블록 추출 const toolCalls = response.content.filter(b => b.type === 'tool_use'); // 3. 도구 호출이 없으면 → 완료 if (toolCalls.length === 0) break; // 4. 각 도구 실행 + 결과를 메시지에 추가 for (const call of toolCalls) { const result = await executeTool(call); messages.push({ role: 'tool_result', content: result }); } // 5. 루프 반복 → LLM이 결과를 보고 다음 행동 결정 }

초기 버전에 있었던 것

  • while(true) Agentic Loop
  • 기본 도구 (검색, 읽기, 쓰기, 실행)
  • 기본 권한 (모든 호출에 사용자 확인)
  • 스트리밍 응답 출력
  • Git 커밋

이후 추가된 것 (전부 피드백에 의해)

  • 4단계 컨텍스트 압축
  • Error Withholding + Cascade Recovery
  • Streaming Tool Executor
  • 5계층 Permission Pipeline
  • Sub-Agent + CacheSafeParams
  • Plugin/Skill/MCP 확장 시스템
  • Auto Mode (ML 분류기)
  • 메모리 시스템 (자동 추출/주입)

1

Agent Skills

Agent Skills: 에이전트에게 전문성을 부여하는 개방형 표준

MCP가 "도구 연결"을 표준화했다면, Skills는 "절차적 지식"을 표준화했다

Skills의 3단계 Progressive Disclosure

L1
메타데이터 (~100 토큰) — 세션 시작 시 name + description만 로드. 에이전트는 스킬의 존재만 인지.
L2
핵심 지시사항 (<5K 토큰) — 작업이 매칭되면 SKILL.md 전체 로드. 구체적 절차 획득.
L3
상세 자료 (무제한) — 실행 중 필요할 때만 scripts/, references/ 로드. 주의력 예산 절약.

"번들링 가능한 컨텍스트의 양은 사실상 무제한" — 필요할 때만 로드하므로

Key Patterns

가장 가치 있는 3가지 구현 패턴

가장 가치 있는 5가지 구현 패턴

패턴 0: Think Tool — 추론 품질 54% 향상 (구현 5줄) ▶

문제: 복잡한 순차 도구 호출에서 모델이 중간에 멈춰서 생각하지 않음. Extended Thinking은 첫 응답 전에만 작동.

해법: 부작용 없는 "think" 도구를 추가. 모델이 도구 호출 사이에 명시적으로 추론 가능.

// 전체 구현 — JSON 5줄 { "name": "think", "description": "Think about something. No side effects.", "input_schema": { "properties": { "thought": { "type": "string" } } } } // 결과: 항공사 도메인 pass^1 0.332 → 0.584 (54% 향상) // SWE-bench: +1.6% (p < .001, d = 1.47)

최적화 프롬프트: "모든 행동 전에 think로: (1) 적용 규칙 나열 (2) 정보 완전성 확인 (3) 정책 준수 검증"

패턴 1: CacheSafeParams — Multi-Agent 비용 문제의 핵심 해법 ▶

문제: Sub-Agent마다 시스템 프롬프트+도구를 새로 전송하면 입력 비용이 에이전트 수에 비례하여 증가

해법: 부모와 자식이 동일한 {시스템 프롬프트, 도구, 모델}을 공유하면 API Prompt Cache 재사용

type CacheSafeParams = { systemPrompt: SystemPrompt // 동일 → 캐시 HIT tools: Tool[] // 동일 → 캐시 HIT model: string // 동일 → 캐시 HIT thinkingConfig: ThinkingConfig baseMessages: Message[] // 접두사 동일 → 캐시 HIT } // 결과: 자식 에이전트 입력 비용 ~90% 절감 // 이것 없이 Multi-Agent는 비용적으로 실행 불가능
패턴 2: Streaming Tool Executor — 레이턴시를 숨기는 기술 ▶

핵심: 모델이 tool_use 블록을 생성하는 즉시 해당 도구의 실행을 시작. 모델 스트리밍 5-30초 윈도우를 활용.

for await (const msg of callModel(...)) { if (msg.type === 'tool_use') { // 스트리밍 중 즉시 도구 실행 시작! streamingExecutor.addTool(msg); } } // 스트리밍 완료 시 대부분의 도구가 이미 완료됨

동시성 규칙:
• 읽기 도구 (Read, Grep): isConcurrencySafe=true → 병렬 실행
• 쓰기 도구 (Edit, Write): isConcurrencySafe=false → 직렬 실행
• Bash 에러: 형제 도구에 캐스케이드 (다른 도구 타입은 독립)

패턴 3: 5계층 Permission Pipeline — Fail-Closed의 코드화 ▶
0
OS 수준 샌드박싱 — 파일시스템 + 네트워크 이중 격리 (Linux: bubblewrap, macOS: seatbelt). 자격증명은 샌드박스 밖에서만 관리. Permission 프롬프트 84% 감소.
1
도구 가시성 제어 — Deny 규칙 도구를 API 호출 전 제거. 모델은 차단된 도구의 존재 자체를 모름.
2
입력 검증 — Zod 스키마 + 도구별 시맨틱 검증 (Bash: AST 파싱)
3
권한 검사 — 도구별 정책 (BashTool: 패턴매칭+AST+시맨틱분석+샌드박싱+ML분류기)
4
Hook 실행 — PreToolUse 사용자 정의 Hook. 입력 수정, 거부, 허용 가능. "defer"로 외부 통합 대기.
5
컨텍스트별 핸들러 — Interactive(CLI), Coordinator(워커), Swarm(팀), SDK(API)

Fail-Closed 기본값: 새 도구는 isConcurrencySafe=false, isReadOnly=false. 개발자가 명시적으로 안전 선언해야 완화.

안전성 연구 데이터: 평가 환경은 실제 위험을 8.5배 과소평가 (55.1% vs 6.5%). AI 코드 리뷰어는 의도적 버그 99% 탐지. Defense-in-depth + 런타임 모니터링 필수.

Design Framework

5가지 Pain Point Transfer 패턴

새로운 기능을 설계할 때마다 이 5가지 질문을 던져라

1. Silent Success

"성공한 작업은 보이지 않아야 한다"

질문: 이 작업이 성공했을 때 사용자에게 알릴 필요가 있는가?
적용: Error Withholding, 자동 메모리, 자동 체크포인트, 캐시 최적화

2. Progressive Learning

"과거 결정에서 미래를 예측한다"

질문: 사용자가 이전에 비슷한 결정을 한 적이 있는가?
적용: Permission 학습, 규칙 추출, 메모리, CLAUDE.md 계층

3. Time Overlap

"대기 시간을 작업 시간으로 변환"

질문: 기다리는 동안 미리 할 수 있는 작업이 있는가?
적용: Parallel Prefetch, Streaming Tool Executor, Memory Prefetch

4. Information Layering

"모든 정보를 보여주는 것 = 아무것도 안 주는 것"

질문: 이 정보를 사용자가 항상 볼 필요가 있는가?
적용: Progressive Disclosure, 결과 접기, 에러 자르기, Ctrl+O 확장

5. Invisible Safety

"보안의 대부분은 사용자에게 보이지 않아야 한다"

질문: 이 보안 조치가 사용자 워크플로우를 중단하는가? → 중단한다면 자동화/은닉 방법을 찾아라.
적용: 도구 가시성 제어, Fail-Closed 기본값, 다층 Bash 보안, ML 분류기, 감사 추적

Implementation

구축 로드맵: Phase 1-4

Phase 1: Foundation

4-6주
  • while(true) Agentic Loop
  • 3-5개 핵심 도구
  • 기본 권한 (모든 도구에 확인)
  • 스트리밍 응답 출력

Phase 2: Resilience

4-6주
  • 컨텍스트 압축 (자동, 2단계+)
  • Error Withholding + Cascade
  • API 재시도 (은닉 + FG/BG 분리)
  • 비용 추적

Phase 3: Scale

6-8주
  • Sub-Agent (CacheSafeParams)
  • MCP 클라이언트 통합
  • Streaming Tool Executor
  • 권한 학습 (규칙 추출)

Phase 4: Autonomy

지속적
  • ML 기반 자동 승인
  • Plugin/Skill 확장 시스템
  • 검증 분리 (Evaluator)
  • OpenTelemetry 관찰가능성
