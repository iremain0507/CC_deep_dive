## 권장 발표 구성: 90분 세션

### Opening (5분): 왜 이 주제가 중요한가

**핵심 메시지를 던져라:**

> "Agent = Model + Harness. 모델은 API 호출이다. 하네스가 제품이다."
> "Meta가 Manus를 $2B에 인수한 것은 모델이 아니라 하네스 때문이었다."

**청중의 관심을 끌 데이터:**
- 92%의 기업이 Agentic AI 비용 초과 (IDC)
- 40%의 multi-agent 파일럿이 6개월 내 실패 (Gartner)
- Google DeepMind: multi-agent는 오류를 17배 증폭
- 그런데 Claude Code는 6개월 만에 $1B 매출을 달성

> "이 간극은 어디서 오는가? 하네스 설계의 차이다."

### Part 1 (15분): 이론적 기반 — "구현 전에 알아야 할 것"

**설명 순서:**

1. **"Building Effective Agents"의 핵심 3가지** (5분)
   - "단순한 것부터 시작하라"
   - Workflow vs Agent 구분
   - 5가지 조합 가능 패턴 (1→5 복잡도순)
   
   **예시 활용**: "여러분이 '고객 문의 자동 분류' 시스템을 만든다면, 이것은 5번(Agent)이 아니라 2번(Routing)으로 충분합니다. 에이전트는 마지막 수단이에요."

2. **Agent = Model + Harness 공식** (5분)
   - OS 비유: LLM=CPU, 컨텍스트=RAM, 하네스=OS
   - 하네스의 7개 서브시스템 개요

3. **MCP가 왜 중요한가** (5분)
   - 도구 통합의 N×1 문제 해결
   - 현재 상태: Linux Foundation 거버넌스, 모든 주요 제공자 채택

**전환 멘트**: "이론은 여기까지입니다. 이제 Claude Code가 이 원칙을 어떻게 코드로 실현했는지, 진화 과정을 따라가 보겠습니다."

---

### Part 2 (10분): 최소 하네스 — "Day 1에 만들 것"

**설명 순서:**

1. **최초의 Agentic Loop** (5분)
   ```
   while (true) {
     API 호출 → tool_use 감지 → 도구 실행 → 결과 주입 → 반복 or 종료
   }
   ```
   
   **핵심 강조**: "이것이 모든 Agentic 시스템의 시작점입니다. Claude Code도 여기서 시작했습니다. Day 1에 40개 도구, 5계층 권한, 4단계 압축을 만들 필요가 없습니다."

2. **"있었던 것 vs 없었던 것" 테이블** (5분)
   - 초기 버전에 있었던 것: 기본 도구, 기본 권한, while(true)
   - 이후 추가된 것: 거의 모든 것
   
   **핵심 강조**: "이 모든 것이 프로덕션 피드백에 의해 추가되었습니다. 미리 만든 게 아니에요."

---

### Part 3 (20분): 프로덕션의 현실 — "사용자가 부딪힌 문제와 해법"

**이 파트가 가장 중요합니다.** 청중이 가장 공감하고, 가장 많이 배우는 구간.

**설명 방식**: 각 Pain Point에 대해 "문제 → 사용자 인용 → 진화 단계 → 최종 해법" 순으로 설명

1. **Permission Pain Point** (7분)
   
   > "매번 허용 눌러야 해요" → 규칙 학습 → 200ms Grace Period → Auto Mode
   
   **데모 가능하면**: 실제로 "npm run build"를 한 번 허용한 후, "npm run test"가 자동 승인되는 것을 보여준다.
   
   **핵심 통찰**: "보안과 사용성은 트레이드오프가 아닙니다. 시스템이 사용자로부터 학습하면 둘 다 올라갑니다."

2. **Error Handling Pain Point** (7분)
   
   > "갑자기 에러 나요" → 3회 은닉 재시도 → Error Withholding → 모델 Fallback
   
   **핵심 통찰**: "에러를 보여줬을 때 사용자가 할 수 있는 조치가 있는가? 없다면 숨기고 자동 복구하세요. 해결책이 없는 정보는 불안만 유발합니다."

3. **Context Window Pain Point** (6분)
   
   > "대화가 길어지면 멍해져" → 수동 /compact → 87% 자동 트리거 → 4단계 계층적 압축
   
   **핵심 통찰**: "사용자에게 '대화를 짧게 유지하세요'라고 말하는 것은 시스템의 한계를 사용자에게 떠넘기는 것입니다."

**전환 멘트**: "이제 이 해법들이 하네스 안에서 어떻게 구조화되는지 보겠습니다."

---

### Part 4 (15분): 하네스의 구조 — "시스템이 복잡성을 흡수하는 방법"

**설명 순서:**

1. **전체 흐름도** (3분)
   - "사용자 입력 → 입력 처리 → Agentic Loop → 결과 반환"
   - 메인 문서의 Phase 0-3 다이어그램 사용

2. **핵심 패턴 3가지** (12분)

   **패턴 A: Error Withholding** (4분)
   ```
   API 에러 → 숨김 → 복구 시도 → 성공하면 에러 폐기
   "복구에 성공한 에러는 존재하지 않았던 것이다"
   ```
   
   **패턴 B: Streaming Tool Executor** (4분)
   ```
   전통적: 스트리밍 완료 → 도구 실행 시작 (직렬)
   Claude Code: 스트리밍 중 도구 실행 시작 (오버랩)
   → 매 턴 2-5초 절감
   ```
   
   **패턴 C: CacheSafeParams** (4분)
   ```
   부모와 자식이 동일한 {시스템 프롬프트, 도구, 모델}을 공유
   → API Prompt Cache 재사용 → 입력 비용 ~90% 절감
   이것 없이 Multi-Agent는 비용적으로 불가능
   ```

## 발표 팁

### Do's

- **구체적 숫자를 사용하라**: "비용 절감"이 아니라 "입력 비용 90% 절감"
- **사용자 인용(Pain Point)으로 공감을 만들어라**: 기술 설명 전에 항상 "사용자가 무엇을 겪었는가"부터
- **진화 순서를 존중하라**: "처음부터 이게 있었다"가 아니라 "이 문제 때문에 이것이 추가되었다"
- **코드 예시보다 아키텍처 다이어그램을 사용하라**: AI 엔지니어에게 코드 한 줄보다 "왜 이 설계인가"가 더 가치 있음
- **Anthropic 블로그를 인용하라**: "Anthropic이 이렇게 말했다"는 권위가 있음

### Don'ts

- **처음부터 7개 서브시스템을 한꺼번에 설명하지 말라** → 진화 순서를 따라가라
- **Claude Code의 코드 구조를 설명하지 말라** → 이것은 레퍼런스 문서의 역할이다. 발표에서는 패턴과 원칙에 집중
- **"이것을 다 구현해야 합니다"라고 말하지 말라** → "Day 1에는 이것만, 나머지는 피드백이 요구할 때"
- **프레임워크 비교에 시간을 쓰지 말라** → "LangGraph vs CrewAI"보다 "하네스의 원칙"이 더 오래 유효

## 핵심 메시지 3가지 (청중이 기억해야 할 것)

### 메시지 1: "Agent = Model + Harness, 하네스가 제품이다"
> 모델은 API로 누구나 접근 가능. 제품의 차별화는 하네스에서 온다.

### 메시지 2: "최소 하네스로 시작하고, Pain Point가 요구할 때만 복잡성을 추가하라"
> Claude Code도 while(true) + 기본 도구로 시작했다. 나머지는 전부 프로덕션 피드백이 추가했다.

### 메시지 3: "사용자의 인지 부하를 시스템의 계산 부하로 변환하라"
> 사용자의 뇌는 비싸다. 시스템의 CPU는 싸다. 모든 설계 결정을 이 렌즈로 검증하라.

## FAQ 대비

**Q: "프레임워크(LangGraph, CrewAI) 안 쓰고 직접 만들어야 하나요?"**
> A: Anthropic의 권고는 "직접 API를 사용하라"입니다. 프레임워크는 추상화가 도움이 되는 경우에만. Claude Code는 프레임워크 없이 직접 구축되었고, 그것이 세밀한 제어를 가능하게 했습니다. 하지만 프로토타이핑에는 CrewAI가 빠르고, 프로덕션에는 LangGraph가 안정적입니다.

**Q: "Multi-Agent가 정말 필요한가요?"**
> A: Anthropic의 데이터: "토큰 사용량이 성능 분산의 80%를 설명한다." 단일 에이전트로 충분하면 그걸 쓰세요. Multi-Agent는 병렬화가 필요하거나, 단일 에이전트의 도구 선택 품질이 한계에 도달했을 때만. Google DeepMind: multi-agent는 오류를 17배 증폭합니다.

**Q: "비용은 어떻게 통제하나요?"**
> A: 3가지 핵심: (1) CacheSafeParams로 캐시 공유 (~90% 절감), (2) Model Routing (보조 작업에 Haiku), (3) Task Budget (에이전트별 USD 상한). Claude Code는 이 3가지로 비용을 예측 가능한 범위로 통제합니다.

**Q: "보안은 어떻게?"**
> A: Claude Code의 5계층 파이프라인: 도구 가시성 제어 → 입력 검증 → 권한 검사 → Hook 실행 → 컨텍스트별 핸들러. 핵심: Fail-Closed 기본값 (새 도구는 기본적으로 안전하지 않음으로 취급).
