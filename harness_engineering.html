Context Management

5단계 컨텍스트 관리 파이프라인

매 Agentic Loop 반복마다 API 호출 전에 순차 실행. 순서에 의미가 있다.

1
Tool Result Budget (query.ts:379, 매 반복)
대형 도구 결과(>50,000자)를 디스크 참조로 교체. 이후 API 호출에서 재전송 방지.
2
Snip (query.ts:401, HISTORY_SNIP)
오래된 메시지 그룹을 경계 마커 기반으로 통째로 제거. 비용 0ms. MC 전에 실행하여 snip된 tool_use_id가 MC에 전달되지 않도록.
3
Microcompact (query.ts:412)
오래된 Read/Bash/Grep 결과를 "[Old tool result cleared]"로 교체. tool_use_id 구조 유지 → 캐시 정렬 보존. 일반 MC + Cached MC(서버측 cache_edits) 두 모드.
4
Context Collapse (query.ts:440, CONTEXT_COLLAPSE)
인접 메시지 그룹을 축약. Auto-compact 전에 실행 — collapse가 충분하면 autocompact 불필요 → 세분화된 컨텍스트 보존 (정보 손실 최소화).
5
Auto-compact (query.ts:453)
Forked Agent가 전체 대화 요약. 트리거: 토큰 > effectiveWindow - 13K. 200K 모델 ≈ 167K(~83.5%). 후처리: 파일 5개 + 스킬 25K + MCP 재주입. Circuit Breaker: 3회 연속 실패 시 중단.
!
Reactive Compact (query.ts:1119, 비상)
API 413 에러 후에만 실행. Error Withholding으로 보류 → Collapse drain → 전체 요약. 성공 시 에러 폐기(사용자 모름), 실패 시 표출. hasAttemptedReactiveCompact=true로 재시도 방지.
Hook Architecture

Hook이 예외 없이 실행되는 이유

Hook은 "선택적 플러그인"이 아니라 도구 실행 파이프라인의 필수 단계다

// 도구 호출의 실제 흐름 (query.ts + hooks/useCanUseTool.tsx) 1. 모델이 tool_use 블록 생성 2. canUseTool() 호출 // hooks/useCanUseTool.tsx └→ executePermissionRequestHooks() // PreToolUse Hook 실행 └→ Hook 결과: allow | deny | ask | defer canUseTool()이 Hook을 건너뛰는 경로는 존재하지 않는다 3. 도구 실행 4. PostToolUse Hook 실행 5. 모든 도구 완료 + 모델이 도구 호출 안 하면: └→ handleStopHooks() // query/stopHooks.ts └→ blocking error 반환 시 → 에러를 user 메시지로 주입 → continue // 모델이 에러를 보고 수정 시도 (= 검증 루프) // 보안: 세션 시작 시 captureHooksConfigSnapshot()으로 설정 스냅샷 // → 세션 중 설정 파일이 변경되어도 숨겨진 Hook 수정 불가능 // Death Spiral 방지: lastMessage.isApiErrorMessage이면 Stop Hook 건너뜀

Tool Execution

StreamingToolExecutor: 스트리밍 중 병렬 도구 실행

모델이 tool_use 블록을 생성하는 즉시 도구 실행 시작 — 응답 스트리밍 완료를 기다리지 않는다

// query.ts:838 — 스트리밍 중 즉시 실행 시작 if (streamingToolExecutor && toolBlock) { streamingToolExecutor.addTool(toolBlock, message) // 즉시! } // query.ts:851 — 완료된 결과를 스트리밍 중에도 yield for (const result of streamingToolExecutor.getCompletedResults()) { yield result }

isConcurrencySafe = true (병렬 가능)

Read, Grep, Glob, WebFetch, WebSearch, LSP
읽기 전용 도구 — 서로 부작용 없음

isConcurrencySafe = false (단독 실행)

Bash, Edit, Write, Agent, 기타
쓰기 도구 — 새 도구는 기본 false (Fail-Closed)

에러 전파: Bash 에러 → siblingAbortController.abort() → 형제 취소. Read/WebFetch 에러 → 독립.
결과 순서: 병렬이더라도 tool_use 순서대로 yield.
Fallback: 스트리밍 실패 시 executor.discard() → 새로 생성 → tombstone 처리.
Memory Prefetch: 동일 패턴 — startRelevantMemoryPrefetch() → Haiku sideQuery → 97% 스트리밍 중 완료.

자동 추출

권한: Read/Grep/Glob 무제한, Write/Edit은 메모리 디렉토리만
상호 배제: hasMemoryWritesSince() — 메인이 직접 기록했으면 포크 건너뜀

매 턴 시작 시 Haiku sideQuery로 관련 메모리 5개 선별

타이밍: 모델 스트리밍과 동시 → 97% 완료 → 추가 대기 0ms
중복 제거: alreadySurfaced 세트로 같은 메모리 재표출 방지

Error Recovery

Error Withholding: 복구에 성공한 에러는 존재하지 않았던 것

// query.ts:800-820 — 스트리밍 중 에러 감지 let withheld = false if (is413(message)) withheld = true // UI에 전달 안 함 if (isMaxTokens(message)) withheld = true if (isMediaSize(message)) withheld = true if (!withheld) yield message // 정상만 통과 // 스트리밍 완료 후 — Cascade Recovery if (withheld413) { collapseDrain() // 1차: 저비용 || reactiveCompact() // 2차: 전체 요약 || surfaceError() // 3차: 포기, 표출 } if (withheldMaxTokens) { escalate(8K → 64K) // 1차 || recoveryMessage() × 3 // 2차: maxOutputTokensRecoveryCount || surfaceError() // 3차: 포기 } // Guard 플래그: 각 경로에 "이미 시도" 플래그 → 무한 루프 방지 // Death Spiral 방지: API 에러 후 Stop Hook 건너뜀
