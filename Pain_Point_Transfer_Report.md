# Pain Point Transfer: 사용자의 고통을 시스템으로 옮기는 기술

## Enterprise Agentic LLM System이 사용자 경험을 설계하는 방법

> **핵심 철학**: 사용자는 "무엇을 할 것인가"에만 집중하고, "어떻게 할 것인가"의 모든 마찰은 시스템이 흡수한다.
>
> **참조 구현체**: Claude Code CLI — ~800K lines, 40+ Tools, 5-Layer Permission, 4-Level Context Compression
>
> **대상 독자**: Enterprise Agentic LLM 시스템을 설계하는 AI 엔지니어

## 1. 설계 원칙: "Silent Success"

Claude Code 전체를 관통하는 UX 설계 원칙은 **"성공적인 작업은 사용자에게 보이지 않아야 한다"**이다.

```
                    Silent Success 원칙
                    
  사용자가 인지해야 하는 것          시스템이 조용히 처리하는 것
  ──────────────────────          ─────────────────────────
  • 내가 요청한 작업의 결과          • API 재시도 (처음 3번은 숨김)
  • 위험한 작업의 확인 요청          • 컨텍스트 압축
  • 최종 비용 요약                  • 토큰 예산 관리
                                   • 메모리 추출/주입
                                   • Prompt Cache 최적화
                                   • 도구 결과 크기 제한
                                   • 권한 학습/자동 승인
                                   • 모델 Fallback
                                   • 세션 상태 영속화
                                   • 프리페치/프리로드
```

이 원칙은 구체적인 코드 패턴으로 구현된다. 각 Pain Point별로 살펴보자.

## 2. Pain Point #1: 반복적 권한 승인

### 사용자의 고통

> "git status 실행할 때마다 '허용'을 눌러야 해요. 같은 명령어인데 왜 매번 물어보죠?"

Agentic 시스템에서 **모든 도구 호출에 사용자 확인을 요구**하면, 10회 도구 호출 시 10번의 중단이 발생한다. 이는 사용자를 "승인 버튼 누르는 기계"로 전락시킨다.

### 시스템이 흡수한 방법

Claude Code는 **4중 자동 승인 레이싱** 아키텍처로 이 문제를 해결한다:

```
┌────────────────────────────────────────────────────────────┐
│           Permission Resolution Race                        │
│                                                            │
│   도구 호출 발생                                             │
│     │                                                      │
│     ├─→ [규칙 엔진] 기존 규칙과 매칭? ──── YES → 즉시 실행    │
│     │                                                      │
│     ├─→ [ML 분류기] 비동기 분류 시작 ─┐                      │
│     │                                │                     │
│     ├─→ [Hook 시스템] PreToolUse ───┐│                     │
│     │                               ││                     │
│     └─→ [사용자 대화상자] 표시 ─────┐││                     │
│                                    │││                     │
│         200ms Grace Period         │││                     │
│         ┌──────────────────┐       │││                     │
│         │ 사용자가 키보드를   │       │││                     │
│         │ 터치하지 않으면...  │       │││                     │
│         └────────┬─────────┘       │││                     │
│                  │                 │││                     │
│         분류기가 먼저 승인하면       │││                     │
│         ✓ 자동 실행 (대화상자 안 보임) │││                    │
│                                    │││                     │
│         누가 먼저든, 첫 응답이 승리   ←┘│                    │
│         (atomic claim() 패턴)        ←┘                    │
│         나머지 경로는 자동 취소        ←┘                    │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 핵심 메커니즘 1: 규칙 학습 (`bashPermissions.ts`)

사용자가 한 번 "허용"하면, 시스템이 자동으로 **재사용 가능한 규칙**을 추출한다:

```
사용자가 "npm run build --prod"를 허용
  │
  ├─ 시스템: 안정적 접두사 "npm run" 추출
  ├─ 제안: "npm run:* 명령어를 항상 허용할까요?"
  ├─ 사용자 수락 → alwaysAllowRules에 Bash(npm run:*) 저장
  │
  └─ 이후 "npm run test", "npm run lint" 등 → 자동 승인
```

**특수 처리**:
- **Heredoc/멀티라인 명령어**: 정확한 매칭이 불가능 → 접두사 규칙으로 자동 격하
- **환경 변수 필터링**: `NODE_ENV=prod npm run build`에서 `NODE_ENV=prod`를 안전하게 제거하여 `Bash(npm run:*)` 규칙과 매칭
- **복합 명령어 제한**: `MAX_SUGGESTED_RULES_FOR_COMPOUND = 5`로 과도한 규칙 제안 방지

#### 핵심 메커니즘 2: 200ms Grace Period (`interactiveHandler.ts`)

```
분류기 비동기 실행 시작 ──→ (50-200ms 소요)
                            │
대화상자 표시 ────────────→  │
                            │
     200ms Grace Period     │
     ┌────────────────┐     │
     │ 사용자가 아직    │     │
     │ 키를 안 눌렀다면 │     ↓
     └────────┬───────┘  분류기 승인 도착
              │              │
              └──── 조건 충족: 자동 실행 ✓
                   대화상자 자동 닫힘

     사용자가 키를 눌렀다면?
     → 분류기 결과 폐기, 사용자 판단 존중
```

#### 핵심 메커니즘 3: 멀티채널 레이싱

권한 요청은 동시에 여러 경로로 전달된다:
- **로컬 CLI** 대화상자
- **Bridge/CCR** (claude.ai 웹 UI)
- **채널 릴레이** (Telegram, iMessage 등)

**첫 번째 응답자가 승리**하며, `atomic claim()` 패턴으로 이중 실행을 방지한다.

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "점진적 신뢰 구축"             │
│                                                         │
│  Phase 1: 모든 것을 물어본다 (신뢰 0%)                    │
│  Phase 2: 사용자가 허용한 패턴을 학습 (신뢰 축적)          │
│  Phase 3: 학습된 패턴은 자동 승인 (마찰 감소)              │
│  Phase 4: ML 분류기가 새로운 패턴도 예측 (마찰 최소화)      │
│                                                         │
│  핵심: 보안을 포기하지 않으면서 마찰을 줄이는 유일한 방법은  │
│       "사용자의 과거 결정에서 미래 결정을 예측"하는 것       │
└─────────────────────────────────────────────────────────┘
```

**구현 체크리스트**:
- [ ] 사용자 결정에서 재사용 가능한 규칙을 자동 추출하는 엔진
- [ ] 비동기 분류기와 대화상자의 레이싱 (Grace Period 포함)
- [ ] 규칙의 세션 간 영속화 (project/user settings)
- [ ] 복합 명령어의 접두사 분해
- [ ] 환경 변수 등 노이즈 제거 후 패턴 매칭

## 3. Pain Point #2: 에러와 불안정성

### 사용자의 고통

> "작업 중에 'API Error 429' 같은 메시지가 뜨면 불안해요. 뭘 해야 하는지도 모르겠고, 작업이 날아간 건지도 모르겠어요."

API 장애, 네트워크 히컵, 토큰 초과 등은 Agentic 시스템에서 일상적이다. 그러나 사용자에게 이런 인프라 수준의 오류를 노출하는 것은 불필요한 불안을 유발한다.

### 시스템이 흡수한 방법

Claude Code의 가장 독창적인 패턴: **Error Withholding** — 복구 가능한 오류를 사용자에게 보여주지 않고, 복구에 성공하면 아예 존재하지 않았던 것처럼 처리한다.

```
┌────────────────────────────────────────────────────────────┐
│              Error Visibility Spectrum                       │
│                                                            │
│  사용자가 절대 보지 못하는 것:                                │
│  ├─ 첫 3번의 API 재시도 (retryAttempt < 4이면 hidden)       │
│  ├─ ECONNRESET/EPIPE → 자동 연결 풀 교체                    │
│  ├─ 413 Prompt-Too-Long → Context Drain/Compact 후 복구     │
│  ├─ Max-Output-Tokens → 상한 확장(8K→64K) 후 계속           │
│  ├─ OAuth 401 → 자동 토큰 갱신 후 재시도                     │
│  ├─ AWS/GCP 자격증명 만료 → 캐시 삭제 + 갱신                 │
│  └─ 429 Rate Limit → Exponential Backoff (배경에서)          │
│                                                            │
│  사용자가 4번째 재시도부터 보는 것:                           │
│  ├─ "Retrying (attempt 4/10)... 15s"                       │
│  ├─ 카운트다운 타이머로 남은 대기 시간 표시                   │
│  └─ API 에러 메시지 (1000자로 자른 후)                       │
│                                                            │
│  사용자가 즉시 보는 것:                                      │
│  ├─ 복구 불가능한 인증 실패                                  │
│  └─ 모든 복구 전략 소진 후의 최종 오류                       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 핵심 메커니즘 1: 3회 은닉 재시도 (`withRetry.ts`)

```typescript
// 외부 빌드에서 처음 3번의 재시도는 완전히 숨김
const hidden = "external" === 'external' && retryAttempt < 4
// → 사용자에게 SystemAPIErrorMessage를 표시하지 않음
// → 일시적 네트워크 히컵은 사용자가 인지조차 못 함
```

**왜 3번인가**: 일반적인 transient 오류(소켓 리셋, 순간 과부하)는 1-2번 재시도로 해결된다. 3번이면 99%+ 의 일시적 오류를 커버하면서도 실제 장애 시 과도한 지연을 피한다.

#### 핵심 메커니즘 2: Foreground/Background 재시도 분리

```
┌─────────────────────────────────────────────────┐
│  529 (Overload) 재시도 전략                       │
│                                                 │
│  Foreground (사용자가 기다리는 중):               │
│  ├─ repl_main_thread → 재시도 O (3회)            │
│  ├─ sdk, agent:* → 재시도 O                      │
│  └─ 이유: 사용자가 대기 중이므로 포기보다 재시도     │
│                                                 │
│  Background (사용자가 모르는 작업):                │
│  ├─ summaries → 즉시 실패 (재시도 X)              │
│  ├─ classifiers → 즉시 실패                      │
│  ├─ title suggestions → 즉시 실패                │
│  └─ 이유: 재시도하면 과부하 증폭 (3-10x)           │
│          사용자가 실패를 알지도 못하는 작업           │
│                                                 │
└─────────────────────────────────────────────────┘
```

#### 핵심 메커니즘 3: 자동 모델 Fallback

```
Opus 529 3회 연속 → 자동으로 Sonnet으로 전환
  ├─ 사용자 경험: "좀 느리네" (품질은 유사)
  ├─ 시스템 로그: tengu_api_opus_fallback_triggered
  └─ 사용자에게 "모델이 변경되었습니다"를 알리지 않음
      (해결 방법이 없는 정보는 불안만 유발)
```

#### 핵심 메커니즘 4: Max-Tokens 자동 복구

```
API 400: "input + max_tokens exceed limit"
  │
  ├─ 시스템: thinkingConfig.budgetTokens 보존하면서 max_tokens 자동 축소
  ├─ 재시도: 조정된 파라미터로 (사용자 모름)
  ├─ 로그: tengu_max_tokens_context_overflow_adjustment
  └─ 사용자 경험: 아무 일도 일어나지 않은 것처럼 응답 도착
```

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "Invisible Recovery"          │
│                                                         │
│  설계 질문: "이 에러를 사용자에게 보여줬을 때,              │
│             사용자가 할 수 있는 조치가 있는가?"             │
│                                                         │
│  YES → 표시 (예: 인증 실패 → 로그인 필요)                 │
│  NO  → 은닉 + 자동 복구 시도                              │
│       (예: 429 → 재시도, 연결 끊김 → 재연결)              │
│                                                         │
│  핵심 원칙:                                              │
│  "해결책이 없는 정보는 불안만 유발한다"                     │
│  "복구에 성공한 오류는 존재하지 않았던 것이다"              │
└─────────────────────────────────────────────────────────┘
```

**구현 체크리스트**:
- [ ] 재시도 은닉 임계값 (N회까지 사용자에게 숨김)
- [ ] Foreground/Background 작업별 차등 재시도 정책
- [ ] 모델 자동 Fallback (capacity 장애 시)
- [ ] 토큰 초과 자동 조정 (thinking budget 보존)
- [ ] 자격증명 만료 자동 갱신
- [ ] 연결 풀 장애 자동 교체 (keep-alive 비활성화)

## 4. Pain Point #3: 대화가 길어지면 성능 저하

### 사용자의 고통

> "처음엔 빠르고 정확했는데, 대화가 길어지니까 점점 느려지고, 아까 한 얘기를 잊어버려요."

이것은 모든 LLM 기반 시스템의 근본적 한계: Context Window의 유한성. 그러나 사용자는 "컨텍스트 윈도우"가 뭔지 알 필요가 없다.

### 시스템이 흡수한 방법

Claude Code는 사용자가 **컨텍스트 관리의 존재 자체를 인지하지 못하도록** 4단계 자동 압축을 수행한다:

```
┌────────────────────────────────────────────────────────────┐
│         Invisible Context Compression Pipeline              │
│                                                            │
│  사용자 관점:                                               │
│  "나는 그냥 계속 대화하고 있다. 시스템이 알아서 한다."         │
│                                                            │
│  시스템 내부:                                               │
│                                                            │
│  매 턴마다:                                                 │
│  ├─ 토큰 계산 → 전체 윈도우의 87% 도달?                     │
│  │   ├─ NO → 아무것도 안 함                                 │
│  │   └─ YES → Auto-Compact 시작                            │
│  │           ├─ Forked Agent가 이전 대화 요약               │
│  │           ├─ 요약 전: 이미지 제거 + [image] 마커 대체     │
│  │           ├─ 요약 후: CompactBoundaryMessage 삽입        │
│  │           ├─ 경계 이전 메시지 → GC 대상                  │
│  │           ├─ 스킬(5개, 각 5K 토큰) + 파일(5개) 재주입     │
│  │           └─ 사용자는 아무것도 모름                       │
│  │                                                        │
│  API 에러 발생 시 (413):                                    │
│  ├─ Error Withholding (사용자에게 안 보여줌)                 │
│  ├─ 1차: Context-Collapse Drain (빠르고 저비용)             │
│  ├─ 2차: Reactive Compact (비상 전체 요약)                  │
│  └─ 성공 → 에러 폐기, 사용자는 모름                         │
│                                                            │
│  안전장치:                                                  │
│  ├─ 연속 실패 3회 → 재시도 중단 (Circuit Breaker)            │
│  ├─ Compact 비용도 Task Budget에 포함                       │
│  └─ 20% 폴백: 파싱 불가 시 가장 오래된 메시지 그룹 제거       │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

#### 사용자가 절대 보지 못하는 세부사항

| 내부 작업 | 사용자 경험 |
|----------|-----------|
| 토큰 87% 도달 → Auto-compact 시작 | 아무 표시 없음 |
| Forked Agent가 3초간 요약 생성 | 아무 표시 없음 (모델 스트리밍 중 실행) |
| 이미지 → `[image]` 마커 대체 | 이전 이미지 참조는 텍스트로 유지 |
| CompactBoundary 삽입 | 아무 표시 없음 |
| 스킬/파일 재주입 (최대 25K 토큰) | 시스템이 여전히 프로젝트 맥락을 기억하는 것처럼 느낌 |

#### Post-Compact 복원의 비밀

압축 후에도 시스템이 "기억하는 것처럼" 느껴지는 이유:

```
압축 후 자동 재주입:
  ├─ 최대 5개 스킬 × 5,000 토큰 = 25,000 토큰
  ├─ 최대 5개 관련 파일 (CLAUDE.md 등)
  ├─ 현재 활성화된 MCP 서버 정보
  └─ 메모리 시스템의 관련 기억 (자동 선별)

→ 사용자: "이 시스템은 절대 잊어버리지 않네"
→ 현실: 세부사항은 사라졌지만, 핵심 맥락은 재주입됨
```

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "Invisible Degradation"       │
│                                                         │
│  사용자가 성능 저하를 느끼지 않게 하려면:                   │
│  1. 능동적 임계값 (87%)에서 미리 압축                     │
│  2. 압축 중에도 핵심 맥락을 재주입                        │
│  3. 압축 실패는 은닉 + Cascade Recovery                  │
│  4. 연속 실패 시 Circuit Breaker로 자원 낭비 방지          │
│                                                         │
│  핵심: "사용자에게 '대화를 짧게 유지하세요'라고 말하는 것은  │
│        시스템의 한계를 사용자에게 떠넘기는 것이다"           │
└─────────────────────────────────────────────────────────┘
```

## 5. Pain Point #4: 시스템이 뭘 하는지 모르겠음

### 사용자의 고통

> "10초째 아무 반응 없이 기다리고 있어요. 뭘 하고 있는 건지, 멈춘 건지 모르겠어요."

### 시스템이 흡수한 방법: Progressive Disclosure

Claude Code는 **"정보의 계층적 공개"** 전략을 사용한다. 모든 것을 보여주는 대신, 중요도에 따라 단계적으로 공개한다.

```
┌──────────────────────────────────────────────────────────┐
│           Progressive Disclosure Layers                    │
│                                                          │
│  Layer 1: 기본 뷰 (항상 보임)                              │
│  ├─ 도구 배지: "Read(5) Grep(3) Bash(2)"                  │
│  ├─ 최근 작업 힌트: "⤿ searchpattern"                     │
│  └─ 최소 700ms 표시 보장 (MIN_HINT_DISPLAY_MS)             │
│                                                          │
│  Layer 2: 상세 뷰 (Verbose 모드 또는 클릭 확장)             │
│  ├─ 각 도구 호출의 개별 결과                                │
│  ├─ 로더 스피너 + 도구별 렌더링                             │
│  └─ Bash: 마지막 N줄, File: 변경 요약                      │
│                                                          │
│  Layer 3: 전체 뷰 (Ctrl+O로 확장)                          │
│  ├─ 잘린 내용의 전체 출력                                  │
│  └─ API 에러의 전체 메시지                                 │
│                                                          │
│  각 레이어는 사용자의 의도적 선택으로만 열림                  │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 실제 구현 패턴

**읽기/검색 작업 자동 접기** (`CollapsedReadSearchContent.tsx`):

```
10개의 파일 읽기 + 5개의 grep 검색이 실행됨

사용자가 보는 것:
  ┌──────────────────────────────────┐
  │ Read(10) Grep(5) ⤿ "auth"       │
  └──────────────────────────────────┘
  (한 줄. 클릭하면 개별 결과 확장)

사용자가 보지 않는 것:
  - 각 파일의 전체 경로와 내용
  - 각 grep의 매칭 라인
  - 도구 실행 순서와 소요 시간
```

**메모리 작업의 배지 표시**:

```
시스템이 자동으로 메모리를 읽고 쓸 때:

사용자가 보는 것:
  ┌──────────────────────────────────┐
  │ Memory(2) ← 자동 메모리 활동 배지  │
  └──────────────────────────────────┘

사용자가 보지 않는 것:
  - 어떤 메모리 파일을 읽었는지
  - Sonnet이 5개의 관련 메모리를 선별한 과정
  - 메모리 중복 제거 로직
```

**도구 결과 자르기 + 확장 힌트**:

```
API 에러가 5,000자:
  ├─ 자동으로 1,000자로 자름 (MAX_API_ERROR_CHARS)
  ├─ "..." 표시기 + "Ctrl+O to expand" 힌트
  └─ 사용자가 필요할 때만 전체 보기

Bash 출력이 100KB:
  ├─ 디스크에 영속화 (사용자 모름)
  ├─ 미리보기 + 파일 경로 참조 전달
  └─ 컨텍스트 폭발 방지 (다음 호출에서 100KB 재전송 방지)
```

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "Information Layering"        │
│                                                         │
│  Level 0: 진행 중 표시 (스피너, 배지) — 항상 보임          │
│  Level 1: 요약 (카운트, 힌트) — 기본 보임                  │
│  Level 2: 상세 (개별 결과) — 요청 시 보임                  │
│  Level 3: 전체 (원본 출력) — 명시적 확장 시 보임            │
│                                                         │
│  핵심: "모든 정보를 보여주는 것"은 "아무 정보도 주지 않는 것" │
│       과 같다. 계층화가 정보의 가치를 만든다.               │
│                                                         │
│  임계값 참고:                                             │
│  - MIN_HINT_DISPLAY_MS = 700 (빠른 작업도 읽을 수 있게)    │
│  - MAX_API_ERROR_CHARS = 1000 (에러 미리보기)             │
│  - maxResultSizeChars = 100KB (디스크 이관 임계값)         │
└─────────────────────────────────────────────────────────┘
```

## 6. Pain Point #5: 느린 응답과 대기 시간

### 사용자의 고통

> "모델이 생각하는 데 20초, 도구 실행에 5초, 또 생각하는 데 15초... 총 40초를 기다렸어요."

### 시스템이 흡수한 방법: 시간을 겹치는 기술

Claude Code는 **"기다리는 시간"을 "일하는 시간"으로 변환**하는 4가지 오버랩 전략을 사용한다:

```
┌──────────────────────────────────────────────────────────────┐
│              Latency Hiding Architecture                       │
│                                                              │
│  Strategy 1: Startup Parallel Prefetch                        │
│  ─────────────────────────────────────                        │
│  프로세스 시작                                                 │
│    ├─ [즉시] MDM 설정 읽기 ───┐                               │
│    ├─ [즉시] Keychain 읽기 ──┤ (병렬, ~65ms)                  │
│    │                         │                                │
│    ├─ [동시에] 모듈 임포트 (~135ms) ← 여기서 프리페치 완료      │
│    │                                                          │
│    ├─ [렌더링] 첫 화면 표시 ← 사용자가 타이핑 시작              │
│    │                                                          │
│    └─ [백그라운드] 사용자가 타이핑하는 5-30초 동안:             │
│        ├─ getUserContext() (CLAUDE.md 탐색)                    │
│        ├─ getSystemContext() (git 상태)                        │
│        ├─ initUser() (프로필)                                  │
│        ├─ countFilesRoundedRg() (3초 타임아웃)                 │
│        ├─ AWS/GCP 자격증명 프리페치                             │
│        └─ 분석 게이트 초기화                                    │
│                                                              │
│  사용자 체감: "즉시 시작됨"                                     │
│  실제: 135ms 임포트 + 사용자 타이핑 시간에 모든 초기화 숨김      │
│                                                              │
│  Strategy 2: Streaming Tool Executor                          │
│  ──────────────────────────────────                           │
│                                                              │
│  전통적:                                                      │
│  [모델 스트리밍 15-20s]→[도구1 3s]→[도구2 2s] = 20-25s        │
│                                                              │
│  Claude Code:                                                 │
│  [모델 스트리밍 15-20s          ]                              │
│         [도구1 실행 3s─완료]                                   │
│              [도구2 실행 2s─완료]                               │
│  = 15-20s (도구 실행 시간이 스트리밍에 숨겨짐)                  │
│                                                              │
│  절감: 2-5초 (매 턴마다)                                       │
│                                                              │
│  Strategy 3: Memory/Skill Prefetch                            │
│  ───────────────────────────────                              │
│                                                              │
│  [모델 스트리밍 시작] → [Memory Prefetch 시작 (Haiku)]         │
│       ...15-20s...      ...3-5s...                            │
│  [모델 스트리밍 완료] → [Memory Prefetch 이미 완료 ✓]          │
│  [도구 실행 완료] → [메모리 결과 소비] ← 0ms 추가 대기          │
│                                                              │
│  97%의 경우 모델 스트리밍 중에 프리페치 완료                     │
│                                                              │
│  Strategy 4: Tool-Use Summary (Off-Thread)                    │
│  ────────────────────────────────────────                     │
│                                                              │
│  [도구 실행 완료] → [Haiku 요약 생성 시작] (비동기, 비차단)     │
│  [다음 API 호출 시작]                                          │
│       ...15s...      → [요약 완료]                             │
│  [다음 API 호출 중] → [요약 결과 소비] ← 0ms 추가 대기          │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

#### 핵심 수치

| 최적화 | 숨겨진 시간 | 체감 효과 |
|--------|-----------|----------|
| Startup Prefetch | ~65ms Keychain + git/context | "즉시 시작" |
| Streaming Tool Executor | 2-5s/턴 | 매 턴 도구 대기 제거 |
| Memory Prefetch | 3-5s/턴 | 메모리 로딩 0ms |
| Tool-Use Summary | 0.5-2s/턴 | 요약 생성 0ms |
| Deferred Heavy Modules | OTel 400KB + gRPC 700KB | 시작 시 1s+ 절약 |

#### 10분 매뉴얼 하우스키핑 지연

```typescript
// 매우 느린 작업은 세션 시작 10분 후 실행
const DELAY = 10 * 60 * 1000  // 10 minutes
setTimeout(runVerySlowOps, DELAY).unref()

// 24시간마다 반복 (npm 캐시 정리, 구버전 제거)
const interval = setInterval(() => {
  void cleanupNpmCacheForAnthropicPackages()
  void cleanupOldVersionsThrottled()
}, 24 * 60 * 60 * 1000)
interval.unref()  // 프로세스 종료를 막지 않음
```

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "Time Overlap"                │
│                                                         │
│  원칙: "사용자가 기다리는 모든 순간은 시스템이 일할 기회"    │
│                                                         │
│  기다리는 시간             숨길 작업                       │
│  ──────────────           ──────────                     │
│  모듈 임포트 (135ms)   →   Keychain + MDM 프리페치        │
│  사용자 타이핑 (5-30s) →   Context/Memory/Analytics 초기화 │
│  모델 스트리밍 (15-20s) →  도구 실행 + 메모리/스킬 프리페치  │
│  다음 API 호출 (15s)   →  도구 요약 생성                   │
│  세션 유지 (10분+)     →  캐시 정리, 버전 관리              │
│                                                         │
│  구현 핵심:                                              │
│  - 프리페치는 "settled되면 소비" 패턴 (절대 차단하지 않음)   │
│  - .unref()로 프로세스 종료를 막지 않는 타이머              │
│  - 3초 타임아웃으로 프리페치가 메인 루프를 지연시키지 않음    │
└─────────────────────────────────────────────────────────┘
```

## 7. Pain Point #6: 매번 같은 지시 반복

### 사용자의 고통

> "매번 '테스트는 jest로 해', '커밋 메시지는 한글로 써' 같은 걸 반복해야 해요."

### 시스템이 흡수한 방법: 3계층 자동 메모리

```
┌──────────────────────────────────────────────────────────┐
│           Automatic Memory Architecture                    │
│                                                          │
│  Layer 1: 자동 추출 (사용자 노력 0)                        │
│  ├─ 대화 종료 시 Forked Agent가 자동으로 메모리 추출          │
│  ├─ 허용 도구: Read/Grep/Glob (읽기), Write/Edit (메모리만) │
│  ├─ 메모리 디렉토리 외 파일 수정 불가 (안전장치)             │
│  ├─ 프롬프트 캐시 공유 → 추가 비용 최소화                   │
│  └─ 사용자는 메모리 추출이 일어났는지조차 모름               │
│                                                          │
│  Layer 2: 자동 주입 (사용자 노력 0)                        │
│  ├─ 매 턴마다 Sonnet이 관련 메모리 5개 자동 선별            │
│  ├─ 이미 표출된 메모리 중복 제거                            │
│  ├─ 현재 사용 중인 도구의 참조 문서 제외 (노이즈 감소)       │
│  ├─ 모델 스트리밍 중 프리페치 (0ms 추가 대기)               │
│  └─ 사용자: "이 시스템은 내 선호를 기억하네"                 │
│                                                          │
│  Layer 3: 팀 동기화 (설정만 하면 자동)                      │
│  ├─ Git remote hash 기반 프로젝트 식별                     │
│  ├─ SHA256 체크섬 기반 델타 동기화                          │
│  ├─ 충돌 시 서버 우선 (pull wins)                          │
│  ├─ 삭제는 전파되지 않음 (안전)                             │
│  └─ 팀원이 저장한 프로젝트 맥락이 자동으로 공유              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

#### 메모리 유형의 닫힌 분류 — "무엇을 기억하고, 무엇을 기억하지 않을 것인가"

```
기억하는 것 (코드에서 도출 불가):        기억하지 않는 것 (코드에서 도출 가능):
  ├─ 사용자 역할과 선호                   ├─ 코드 패턴과 규약
  ├─ 접근 방식 교정/확인                  ├─ Git 히스토리
  ├─ 프로젝트 마감, 결정, 사건            ├─ 디버깅 해결책
  └─ 외부 시스템 위치                     └─ CLAUDE.md에 이미 있는 내용
```

이 구분이 중요한 이유: **무엇이든 기억하면 메모리가 노이즈로 가득 차서** Context Distraction을 유발한다. 코드에서 `git log`로 알 수 있는 것을 메모리에 저장하면, 메모리 용량만 낭비된다.

### 엔지니어를 위한 설계 인사이트

```
┌─────────────────────────────────────────────────────────┐
│  Pain Point Transfer 패턴: "Invisible Learning"          │
│                                                         │
│  1. 추출은 자동 (사용자가 "기억해"라고 말할 필요 없음)      │
│  2. 주입은 선별적 (5개만, 관련성 기반)                     │
│  3. 저장 범위는 제한적 (코드에서 도출 가능한 것은 제외)      │
│  4. 비용은 최소화 (프롬프트 캐시 공유)                     │
│                                                         │
│  핵심 위험: Memory Poisoning (OWASP ASI06)               │
│  대응: 메모리 쓰기는 메모리 디렉토리로 제한                 │
│       닫힌 유형 분류로 노이즈 축적 방지                    │
│       팀 동기화에서 삭제 비전파 (안전)                     │
└─────────────────────────────────────────────────────────┘
```

## 8. Pain Point #7: 프로젝트마다 다른 맥락

### 사용자의 고통

> "React 프로젝트에서 하던 대로 Go 프로젝트에서도 동작해요. 프로젝트마다 다른 규약이 있는데..."

### 시스템이 흡수한 방법: 계층적 컨텍스트 자동 수집

```
┌──────────────────────────────────────────────────────────┐
│        Project-Adaptive Context Hierarchy                  │
│                                                          │
│  Priority 1 (최고): 관리 설정                              │
│  └─ /etc/claude-code/CLAUDE.md (시스템 관리자 규칙)         │
│                                                          │
│  Priority 2: 개인 글로벌 설정                              │
│  └─ ~/.claude/CLAUDE.md (모든 프로젝트 공통 선호)           │
│                                                          │
│  Priority 3: 프로젝트 설정 (코드베이스에 체크인됨)            │
│  ├─ CLAUDE.md                                            │
│  ├─ .claude/CLAUDE.md                                    │
│  └─ .claude/rules/*.md (여러 규칙 파일 조합)               │
│                                                          │
│  Priority 4: 개인 프로젝트 설정 (체크인 안 됨)              │
│  └─ CLAUDE.local.md (gitignore 대상)                      │
│                                                          │
│  자동 수집:                                                │
│  ├─ CWD에서 루트까지 상향 탐색                              │
│  ├─ @include 지시어로 외부 파일 참조                        │
│  ├─ 순환 참조 감지 (무한 루프 방지)                          │
│  ├─ 파일당 40KB 상한 (경고 포함)                            │
│  └─ 모든 수집은 세션 시작 시 자동 (사용자 작업 불필요)        │
│                                                          │
│  + Git 맥락 자동 주입:                                     │
│  ├─ 현재 브랜치, 기본 브랜치                                │
│  ├─ 최근 5개 커밋                                         │
│  ├─ 변경 상태 (2000자로 자름)                              │
│  └─ Git 사용자 이름                                       │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**사용자 경험**: 프로젝트 디렉토리에 들어가서 Claude Code를 실행하면, 시스템이 자동으로:
1. CLAUDE.md 계층을 발견하여 프로젝트 규약 학습
2. Git 상태를 스냅샷하여 현재 작업 맥락 파악
3. 메모리에서 이 프로젝트의 과거 맥락 로드
4. **Agent Skills**에서 프로젝트에 적합한 스킬을 자동 발견 (3단계 Progressive Disclosure: 메타데이터만 ~100 토큰 → 매칭 시 전체 로드 → 실행 시 상세 자료)
5. 이 모든 것을 시스템 프롬프트에 주입

사용자는 "이 프로젝트는 Jest를 쓰고, 커밋은 Conventional Commits를 따르고..." 같은 설명을 할 필요가 없다.

**Agent Skills의 Pain Point 흡수**: Skills는 "에이전트가 특정 작업을 어떻게 수행하는지"를 절차적 지식으로 캡슐화한다. 예를 들어 Excel 스킬은 수식 포함 스프레드시트를, PDF 스킬은 양식 작성을 가능하게 한다. 이 지식은 개방형 표준(agentskills.io)으로 33+ 플랫폼과 호환되며, 조직 수준에서 전사 배포가 가능하다. Skills 2.0은 Evals(ON/OFF 비교), Blind A/B 테스트, Description 최적화(~20회 반복)를 추가하여 "스킬이 제대로 작동하는지 추측이 아닌 측정"을 가능하게 했다.

## 10. Pain Point #9: 세션 단절

### 사용자의 고통

> "1시간 동안 작업했는데 터미널이 닫혔어요. 모든 맥락이 사라졌어요."

### 시스템이 흡수한 방법

```
┌──────────────────────────────────────────────────────────┐
│           Session Continuity Architecture                   │
│                                                          │
│  자동 저장 (사용자 작업 불필요):                             │
│  ├─ 모든 대화 → history.jsonl (JSONL 스트리밍 기록)          │
│  ├─ 대형 붙여넣기 → SHA256 해시 기반 외부 저장소             │
│  │  ├─ ≤1KB → 인라인 저장 (history.jsonl 내)               │
│  │  └─ >1KB → 해시 참조 + 비동기 디스크 기록                 │
│  ├─ 파일 히스토리 체크포인트 (매 어시스턴트 메시지마다)        │
│  │  ├─ 변경 전 파일 내용 자동 캡처                           │
│  │  ├─ 버전 넘버링 (v1, v2, v3..., 최대 100개)              │
│  │  └─ .claude/.file-history/에 해시 기반 저장               │
│  ├─ 비용 상태 → 프로젝트 설정에 영속화                       │
│  └─ Graceful Shutdown → 보류 I/O 완료 대기 후 종료           │
│                                                          │
│  복원 (/resume):                                           │
│  ├─ 같은 레포 세션 우선 로드                                │
│  ├─ 메시지 + 파일 히스토리 + 컨텐츠 교체 상태 복원           │
│  ├─ PR 번호로 필터링 가능                                   │
│  ├─ Git 맥락 + 모델 상태 재설정                             │
│  └─ 비용 데이터 복원 (세션 ID 매칭)                          │
│                                                          │
│  세션 백그라운딩 (Ctrl+B):                                  │
│  ├─ 현재 쿼리를 백그라운드로 전환                            │
│  ├─ 메시지 + 로딩 상태 양방향 동기화                         │
│  └─ 완료 시 메인 UI에 자동 반영                             │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**파일 히스토리의 사용자 경험**:
```
사용자: "아까 수정한 거 되돌려줘"
시스템: (자동 체크포인트에서 이전 버전 복원)

이것이 가능한 이유:
  - 매 어시스턴트 메시지마다 자동 체크포인트 생성
  - .claude/.file-history/에 100개까지 버전 보관
  - 사용자가 "체크포인트 생성"을 요청한 적 없음
```

## 12. Pain Point #11: 보안 불안

### 사용자의 고통

> "AI가 내 파일을 마음대로 수정하거나, 위험한 명령어를 실행할까 봐 걱정돼요."

### 시스템이 흡수한 방법: Invisible Safety Net

보안은 사용자를 **방해하지 않으면서 보호**해야 한다. Claude Code의 접근:

```
┌──────────────────────────────────────────────────────────┐
│           Invisible Safety Architecture                    │
│                                                          │
│  사용자가 느끼는 것:                                       │
│  "위험한 것만 물어보고, 나머지는 알아서 안전하게 해"          │
│                                                          │
│  실제 동작:                                               │
│                                                          │
│  1. 도구 가시성 제어 (사용자 모름)                          │
│  ├─ deny 규칙 도구 → API 호출 전 제거                      │
│  ├─ MCP 서버 차단 → 전체 서버 도구 제거                     │
│  └─ 모델은 차단된 도구의 존재 자체를 모름                    │
│                                                          │
│  2. Fail-Closed 기본값 (사용자 모름)                       │
│  ├─ 새 도구 → 병렬 불안전, 쓰기로 간주                     │
│  ├─ 불확실한 Bash 명령어 → 거부                            │
│  └─ 개발자가 명시적으로 안전 선언해야 완화                   │
│                                                          │
│  3. 다층 Bash 보안 (사용자 모름)                           │
│  ├─ AST 파싱 → 위험 패턴 탐지                              │
│  ├─ 시맨틱 분석 → 명령어 의미 해석                          │
│  ├─ 샌드박싱 → cwd 외 쓰기 격리                            │
│  └─ ML 분류기 → 복잡한 명령어 자동 판단                     │
│                                                          │
│  4. 사용자가 느끼는 것 (최소한의 마찰)                       │
│  ├─ 위험 작업만 확인 요청 (rm, chmod, 외부 전송)             │
│  ├─ 안전한 작업은 자동 승인 (ls, git status, cat)           │
│  └─ 학습된 규칙은 다시 물어보지 않음                        │
│                                                          │
│  5. 감사 추적 (사용자 모름, 관리자용)                        │
│  ├─ 모든 권한 결정 → toolDecisions Map                     │
│  ├─ OTel 추적: 결정/소스/도구명/대기시간                    │
│  └─ EU AI Act 규정 준수 기반                               │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**핵심 통찰**: 보안의 대부분은 사용자에게 보이지 않는다. 사용자는 "가끔 위험한 작업에서 확인 요청이 뜬다" 정도만 인지한다. 실제로는 매 도구 호출마다 5계층 검증이 실행되고 있다.

```
│  ├─ 이미 신뢰됨 → 대화상자 임포트 자체를 건너뜀             │
│  ├─ 빈 디렉토리 체크: 메모이제이션 (반복 FS 히트 방지)       │
│  └─ --bare 모드: 모든 프리페치 건너뜀 (스크립팅 용)          │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

## 14. 종합: Pain Point Transfer 설계 프레임워크

### 12개 Pain Point의 Transfer 패턴 분류

모든 Pain Point Transfer는 **5가지 핵심 패턴**으로 분류된다:

```
┌───────────────────────────────────────────────────────────────┐
│                                                               │
│  패턴 1: Silent Success (조용한 성공)                          │
│  ─────────────────────────────────                            │
│  "성공한 작업은 보이지 않아야 한다"                              │
│                                                               │
│  적용: Error Withholding, 자동 메모리, 자동 체크포인트,          │
│       자동 컨텍스트 압축, 자동 캐시 최적화                       │
│                                                               │
│  구현 질문: "이 작업이 성공했을 때 사용자에게 알릴 필요가 있는가?" │
│  대부분의 답: NO                                               │
│                                                               │
│                                                               │
│  패턴 2: Progressive Learning (점진적 학습)                    │
│  ─────────────────────────────────────                        │
│  "과거 결정에서 미래 결정을 예측한다"                             │
│                                                               │
│  적용: Permission Learning, 규칙 추출, 메모리 축적,             │
│       프로젝트 적응, CLAUDE.md 계층                             │
│                                                               │
│  구현 질문: "사용자가 이전에 비슷한 결정을 한 적이 있는가?"       │
│  있다면: 자동 적용                                             │
│                                                               │
│                                                               │
│  패턴 3: Time Overlap (시간 겹치기)                            │
│  ─────────────────────────────────                            │
│  "사용자가 기다리는 모든 순간은 시스템이 일할 기회"               │
│                                                               │
│  적용: Parallel Prefetch, Streaming Tool Executor,            │
│       Memory Prefetch, Tool-Use Summary, Background Tasks     │
│                                                               │
│  구현 질문: "사용자가 기다리는 동안 미리 할 수 있는 작업이 있는가?"│
│  있다면: 프리페치 + settled되면 소비                             │
│                                                               │
│                                                               │
│  패턴 4: Information Layering (정보 계층화)                    │
│  ────────────────────────────────────                         │
│  "모든 정보를 보여주는 것은 아무 정보도 주지 않는 것"             │
│                                                               │
│  적용: Progressive Disclosure, Result Folding, 에러 자르기,    │
│       배지 시스템, Ctrl+O 확장                                 │
│                                                               │
│  구현 질문: "이 정보의 중요도는 몇 레벨인가?"                   │
│  Level 0: 항상 보임  Level 1: 기본  Level 2: 요청 시  Level 3: 명시적│
│                                                               │
│                                                               │
│  패턴 5: Invisible Safety (보이지 않는 안전)                   │
│  ──────────────────────────────────                           │
│  "보안의 대부분은 사용자에게 보이지 않아야 한다"                  │
│                                                               │
│  적용: 도구 가시성 제어, Fail-Closed 기본값, 다층 Bash 보안,    │
│       감사 추적, 샌드박싱                                      │
│                                                               │
│  구현 질문: "이 보안 조치가 사용자 워크플로우를 중단하는가?"      │
│  중단한다면: 자동화/은닉 방법을 찾아라                           │
│  찾을 수 없다면: 최소한의 마찰로 확인 요청                       │
│                                                               │
└───────────────────────────────────────────────────────────────┘
```

### Pain Point Transfer 설계 체크리스트

새로운 기능을 설계할 때, 각 사용자 접점에서 이 질문을 던져라:

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  □ 이 작업이 성공했을 때 사용자에게 알릴 필요가 있는가?        │
│    → NO면 Silent Success 패턴 적용                          │
│                                                             │
│  □ 사용자가 이전에 비슷한 결정을 한 적이 있는가?               │
│    → YES면 Progressive Learning 패턴 적용                   │
│                                                             │
│  □ 사용자가 기다리는 동안 미리 할 수 있는 작업이 있는가?        │
│    → YES면 Time Overlap 패턴 적용                           │
│                                                             │
│  □ 이 정보를 사용자가 항상 볼 필요가 있는가?                   │
│    → NO면 Information Layering 패턴 적용                    │
│                                                             │
│  □ 이 보안 조치가 사용자 워크플로우를 중단하는가?               │
│    → YES면 Invisible Safety 패턴으로 자동화/은닉             │
│                                                             │
│  □ 이 에러를 사용자에게 보여줬을 때 사용자가 할 수 있는 것이?   │
│    → 없으면 은닉 + 자동 복구                                │
│                                                             │
│  □ 이 설정을 사용자가 매번 해야 하는가?                        │
│    → YES면 자동 감지 + 1회성 확인으로 전환                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
